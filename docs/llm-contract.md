# Контракт LLM-слоя

LLM — не часть детерминированного ядра симуляции. Экономика, цены, труд,
события и save/load остаются в коде. LLM может только смотреть на выданный
snapshot, предлагать действия или текст, а игровой слой валидирует результат
и применяет его через обычные публичные API.

Главный вектор: LLM должен быть честным конкурентом игрока, а не классическим
всевидящим игровым ИИ с бонусами. Сложность конкурента задаётся качеством
модели, prompt-версией и глубиной её анализа, а не скрытыми ресурсами.

## Роли LLM

### 1. Competitor

Будущий основной режим: LLM управляет домом-конкурентом как игрок.

- Видит только observation: ту же информацию, которую человек может увидеть в
  интерфейсе.
- Не получает прямой доступ к объектам `Economy`, скрытым полям, будущим
  событиям, seed RNG или внутренним эвристикам.
- Возвращает список намерений/команд, а не правки состояния.
- Все команды проходят валидацию теми же методами, что и действия игрока.
- Невалидная команда отклоняется и логируется без частичного применения.

### 2. Event proposer

Будущий режим генерации ситуационных событий из текущей экономики и
исторического контекста.

- Модель предлагает событие как кандидата, а не сразу меняет канон игры.
- Кандидат сохраняется с provenance: модель, prompt-версия, observation hash,
  сырой ответ, нормализованное событие, статус `draft/approved/rejected`.
- В общую базу событие попадает только после ревью или явного утверждения.
- Эффекты события должны быть выбраны из разрешённого набора эффектов; модель
  не придумывает новые механики, товары, штрафы или ресурсы.

### 3. Narrative renderer

Текущий v0-режим: `game/openrouter_npc.gd` получает `Gameplay.to_llm_context()`
и возвращает текст сцены поверх уже готового события.

- Это декоративный слой: варианты выбора и эффекты остаются в
  `sim/event_catalog.gd`.
- Если ключа, сети или ответа модели нет, игра остаётся на статическом `body`.
- Ответ: обычный русский текст, 2–4 коротких предложения, без JSON, списков и
  пересказа внутренних численных эффектов.

## Observation contract

Observation для LLM-конкурента должен быть подмножеством того, что видит
игрок:

- текущий тик и год/фаза сценария;
- деньги, связи с казной и публичные ограничения своего дома;
- известные узлы, запасы, целевые запасы, цены, спрос и рынок труда;
- свои предприятия, рецепты, мощности, ставки и работников;
- свои караваны и публично видимые чужие караваны, если они есть в UI;
- открытые контракты, активные контракты своего дома и публичные результаты;
- текущие исторические и игровые события;
- последние сообщения журнала;
- исторический контекст из `docs/historical-context.md`.

Запрещено передавать:

- будущие события и будущие броски RNG;
- скрытые планы других агентов;
- точные приватные состояния конкурентов, если игрок не видит их в UI;
- внутренние коэффициенты, если они не раскрыты игроку;
- прямые ссылки на изменяемые объекты симуляции.

Текущий `Gameplay.to_llm_context()` — v0 snapshot для narrative renderer.
Когда появится LLM-конкурент, нужен отдельный `to_agent_observation(agent)`,
не расширение декоративного snapshot.

## Action contract

LLM-конкурент возвращает строгий JSON с кратким объяснением и списком команд:

```json
{
  "reasoning_summary": "Железо выгоднее отправить в Москву, а Невьянску нужно зерно.",
  "actions": [
    {
      "type": "dispatch",
      "from": "Невьянск",
      "to": "Москва",
      "good": "Железо",
      "qty": 4,
      "sell_on_arrival": true
    }
  ]
}
```

Разрешённые команды должны соответствовать публичным API экономики:

- покупка/продажа товара в известном узле;
- отправка каравана;
- принятие или отказ от контракта;
- изменение ставки наёмным;
- запрос приписных работников;
- строительство или расширение, когда механика появится.

Игровой слой обязан:

- проверять JSON-схему и типы;
- ограничивать число команд за тик;
- применять команды только через API `Economy`/`Gameplay`;
- логировать отклонённые команды с причиной;
- не давать модели компенсаций за невалидный ответ.

## События

Событийная система должна разделять три слоя:

- `historical_events` — ручные детерминированные события эпохи: война,
  казённые заказы, указы, проверки, смена власти, запуск или передача заводов.
- `authored_events` — ручные игровые события с условиями, выбором и эффектами.
- `generated_event_candidates` — LLM-кандидаты, сохранённые для будущего
  использования после ревью.

Исторические события создают окружение и не зависят от модели. LLM может
помочь предложить формулировку или вариант сцены, но не решает, что такое
событие произошло.

## Исторический и стилевой контракт

Источник эпохи и тона: `docs/historical-context.md`.

- Период: петровская и раннепослепетровская Россия, примерно 1700–1745.
- Игрок и конкуренты вдохновлены домами промышленников эпохи, но не обязаны
  буквально повторять биографии Никиты Демидовича или Акинфия Никитича.
- Тон: жёсткий деловой XVIII век без фольклорной клюквы и тяжёлого
  канцелярита.
- Люди говорят о деньгах, сроках, поставках, работниках, связях и риске
  казённой проверки.
- Запрещено уводить базовый контекст в поздний XVIII век и екатерининскую
  эпоху без отдельного решения.

## Детерминизм и replay

Экономика остаётся детерминированной. LLM-решения считаются внешними
действиями, как ввод игрока.

Для воспроизводимости партии нужно сохранять:

- model;
- prompt_version;
- observation hash;
- сырой ответ модели;
- нормализованные команды;
- фактически применённые команды;
- причины отклонения команд.

Replay не должен заново вызывать модель. Он повторяет action-log поверх
детерминированной симуляции.

## Запреты

LLM не имеет права:

- менять деньги, товары, работников, отношения или сроки напрямую;
- создавать новые эффекты вне разрешённой схемы;
- получать скрытую информацию или знание будущего;
- получать бонусы к деньгам, производству, ценам или логистике;
- называть игроку JSON-поля, параметры и внутренние коэффициенты;
- объявлять исход выбора до применения эффекта кодом;
- добавлять исторические факты как канон без проверки;
- превращать посессионных работников в обычный товар в тексте или механике.

## Конфигурация v0

Локальная конфигурация текущего OpenRouter renderer читается из `.env` по
образцу `.env.example`:

```bash
OPENROUTER_API_KEY=
OPENROUTER_MODEL=openai/gpt-oss-20b:free
```

Smoke-test клиента: `llm_smoke.gd` (см. команды в `CLAUDE.md`).
